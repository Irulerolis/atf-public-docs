ATF Archive Reference

This document is the internal reference companion to the operator guide. It is for understanding how the archive works, what files and state exist, and which pieces matter when debugging or extending the system.

No live credentials, session values, browser-exported auth details, or machine-specific secrets are documented here. This is architecture and state reference, not an auth manual.

2. Crawl Pipeline

local config | v test_login.py | v scouts -> queue/report files -> crawler families | v temp files + manifests + progress + SQLite tracking | v MEGA / local archive output | v search, verification, repair, replica work

Important practical point

The pipeline is not one universal “source chain.” Different page types and tools read from different layers. Forum crawling, media crawling, local browsing, and the replica each consume overlapping but not identical state.

Identity rule of thumb

For archival museum work, raw files alone are not enough. The useful identity is normally some combination of url, type, thread_id, thread_page, post_id, mega_path, and content_hash. The exact tuple depends on the artifact type.

Folder naming

  • Forward slashes in names are normalized so path segments do not explode.
  • Illegal filename characters are stripped.
  • Scout-created structure is preferred over ad hoc path guessing later.

Asset manifests

Each crawled thread produces an asset_manifest_*.json file that maps source URLs to local file paths. The replica uses these to serve thread images and attachments via the /forum/archive-asset route. Manifest records include: source URL, local filename, content hash (SHA-256), thread/post placement context, and file type.

5a. The forum/ Package

The forum/ directory is a Python package containing shared modules used by all crawler families. Understanding this package is essential for debugging or extending any crawler.

ModuleRole
tracking.pySQLite-backed crawl state tracking: thread status, URL dedup, subforum progress, error logging. WAL mode for concurrent access.
scrapers.pyPost/thread HTML parsing, content extraction, metadata collection from XenForo pages.

migrate.pyMigration from legacy JSON-based state to SQLite. Safe to re-run (INSERT OR IGNORE).

navigation.pyXenForo page navigation parsing, pagination detection, thread/subforum URL construction.

main.pyCLI entry point and argument parsing for the forum crawler.

5b. Search Database Schema (search.db)

The search database uses SQLite FTS5 for full-text search. It is built by build_search.py from archived thread JSON files, and is also written to live by crawler instances during crawling.

ID type mismatch: search.db stores thread and post IDs as TEXT, while crawl_state.db stores them as INTEGER. When joining across databases or doing lookups in replica/data.py, cast appropriately.

8. Practical Caveats

  • Do not treat the archive browser, saved HTML, and replica as consuming exactly the same source graph.
  • Do not assume a placeholder UI means missing data; sometimes it means missing wiring.
  • Do not overwrite raw source artifacts just because a transformed version looks cleaner.
  • Do not assume queue/state files are the only truth once SQLite tracking is in play.
  • Do not confuse root app.py (operator dashboard) with replica/app.py (Flask replica server).
  • When looking up post IDs across databases, remember: search.db uses TEXT IDs, crawl_state.db uses INTEGER IDs.
