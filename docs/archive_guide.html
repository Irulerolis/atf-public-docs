1. Start Here

The ATF Archive System is a preservation stack for four jobs: mapping the site, crawling forum text, collecting thread media, and mirroring gallery/resources. This guide now puts operator workflow first so you can quickly answer: what should I run next?

How to use this guide: If you are actively running the archive, stay in this file. If you want internals or replica-builder theory, jump to archive_reference.html or archival_museum_design.html.
Security note: This guide intentionally avoids real credentials, session values, browser-exported auth details, and machine-specific secrets. Keep all authentication and storage access details in your local config.py and runtime files only.

3. First Setup

Prerequisites

  • Python 3.11 or 3.12
  • Playwright + Chromium
  • mega.py, requests, and beautifulsoup4
  • customtkinter if you want the GUI launcher / dashboard

Local Configuration

Edit config.py with your local access details and destination settings. Keep real auth/session values in that local file only; the guide deliberately stays generic.

# 1. Verify local access before a long crawl
python test_login.py

# 2. Only after login looks healthy, continue into the scouts

Bring-Up Order

  1. Verify login — python test_login.py
  2. Map the forum — python Scout.py
  3. Map the media gallery — python scout_media.py
  4. Map the resource manager — python scout_resources.py
  5. Start the forum launcher / dashboard — python app.py
  6. Optionally start the gallery and resource crawlers in separate terminals

4. Scouts (Pre-Flight)

4a. Forum Scout (Scout.py)

The forum scout is the first thing to run. It discovers the complete forum hierarchy using XenForo breadcrumbs (more reliable than HTML structure guessing). Note: the filename is Scout.py with a capital S.

What it does:

  1. Starts a Playwright session using your local auth/session settings
  2. Navigates to the forum index page
  3. Collects all forum/subforum URLs
  4. Visits each forum, reads the breadcrumb trail to build the hierarchy
  5. Outputs subforum_map.json (142 subforums with URL, name, description, MEGA path parts)
  6. Pre-creates all MEGA folders before crawling starts

Output: subforum_map.json

Structure: Array of objects with url, name, description, and mega_path_parts (array of folder names for MEGA hierarchy).

4b. Media Scout (scout_media.py)

Maps the XFMG (XenForo Media Gallery) structure for the media crawler.

What it does:

  1. Discovers all XFMG categories
  2. Counts items and pages per category
  3. Samples 10 items per category for overlap analysis
  4. Detects album structures
  5. Identifies AI-generated categories to exclude: ai.20, ai-loli.21, ai-shota.23

Output: media_gallery_report.json

Used by media_queue.py to build the work queue.

4c. Resource Scout (scout_resources.py)

Maps the XFRM (XenForo Resource Manager) structure.

What it does:

  1. Discovers all 86 XFRM categories
  2. Counts resources and pages per category
  3. Samples 10 resources per category
  4. Identifies file-downloadable vs external-only resources
  5. No categories excluded (unlike media)

Output: resources_report.json

Used by resource_queue.py to build the work queue.

5. Forum Crawlers

How Multi-Instance Works

A single crawler.py file supports multiple parallel instances via the --id flag. Each instance gets its own state files, temp directory, and log output. All instances compete for work from the same work_queue.json.

CLI Reference

python crawler.py --id 1 --start 1 --end 50         # crawl subforums 1-50
python crawler.py --id 2 --start 51 --end 100        # crawl subforums 51-100
python crawler.py --id 1 --text-only                 # text + metadata only, queue media URLs
python crawler.py -i 3 -s 1 -e 999 -t               # short flags
FlagShortDefaultDescription
--id-i1Instance ID (1-20). Determines state file names.
--start-s1Start subforum index (1-based, inclusive).
--end-e999End subforum index (inclusive, 999 = no limit).
--text-only-toffSkip all binary downloads. Queue media URLs to pending_media_XX.json for later processing by thread_media_crawler.py.

The dashboard (app.py) launches multiple instances automatically and passes the --id, --start, and --end flags.

Work Queue System (work_queue.py)

The work queue uses dynamic claiming instead of permanent assignment. Key functions:

  • claim_next_subforum() — atomically claims an unclaimed subforum
  • heartbeat_subforum() — proves the instance is still alive (every 5 min)
  • complete_subforum() — marks a subforum as done

Stale timeout: 20 minutes. If a crawler dies, its work becomes available for others.

File locking: Atomic O_CREAT | O_EXCL with a 10-second timeout.

Thread/Post Scraping Flow

  1. Claim a subforum from the work queue
  2. Navigate to subforum listing, discover all thread URLs
  3. For each thread: check crawl_state.db (microsecond SQLite lookup) — skip if done
  4. Mark thread as in_progress in SQLite with instance ownership
  5. Scrape all pages of posts, updating page progress in SQLite
  6. For each post: download embedded images and attachments
  7. Upload files to MEGA in the correct subforum folder, record URLs in SQLite
  8. Index thread/post text into SQLite FTS5 search database
  9. Save thread data as thread.json with all post content
  10. Mark thread as done in SQLite + write legacy .done marker

Search Database (search_db.py)

Each crawler instance writes to the same search.db SQLite database using WAL mode and a 10-second busy timeout for safe concurrent writes from 15+ instances. Uses FTS5 for full-text search across threads, posts, tags, and subforums.

6. Media Gallery Crawler

Overview

media_crawler.py crawls the XenForo Media Gallery (XFMG) — images, videos, and audio files uploaded by users. It excludes AI-generated content from 3 categories (~437K items skipped): ai.20, ai-loli.21, ai-shota.23.

CLI Reference

python media_crawler.py --id 1      # run as instance 1
python media_crawler.py --id 2      # run a second instance in another terminal
FlagShortRequiredDescription
--id-iYesInstance ID (1-based). Determines state file names and queue claiming.

5 Download Strategies (Priority Order)

#StrategyHow It Works
1HTML ParseScans detail page HTML for /data/xfmg/image/, /data/xfmg/video/, or /data/xfmg/audio/ paths in media content containers.
2Thumbnail → Full-ResTransforms thumbnail URL: /data/xfmg/thumbnail/ → /data/xfmg/image/ (XFMG stores them in parallel directories).
3Playwright JSQueries live DOM: img[src*="/data/xfmg/image/"], video source, audio source. Handles lazy-loaded content (data-src).
4XFMG /full EndpointXenForo native full-resolution download: {media_url}/full. Works reliably for videos, returns HTML for images.
5Thumbnail FallbackLast resort — uses the thumbnail URL directly. Lower resolution, logged as a warning.

Album Handling

The crawler detects album pages (URLs matching media/albums/[slug].[id]) on the first listing page. Each album is crawled independently: all pages are enumerated and every item inside is processed via process_item().

Background Thread Pool

The media crawler uses the Phase 1 / Phase 2 split:

  • Phase 1 (main Playwright thread): Navigate to detail page, parse HTML, gather all candidate download URLs from the 5 strategies, mark as in-flight, submit to thread pool, move to next item immediately.
  • Phase 2 (background worker): Try candidate URLs in priority order, download file with streaming SHA-256, rename, upload to MEGA, update manifest, write .done marker.

Queue Chunking (media_queue.py)

Categories are split into 100-page chunks (e.g., "Loli pages 1-100", "Loli pages 101-200"). This balances work across instances since categories vary from 16 to 6,325 pages. Chunking is managed by media_queue.py with the same lease-based claiming as the forum queue.

7. Resource Crawler

Overview

resource_crawler.py crawls the XenForo Resource Manager (XFRM) — downloadable files like game mods, doujins, VAM content, and more across 86 categories. Unlike the media crawler, no categories are excluded.

CLI Reference

python resource_crawler.py --id 1   # run as instance 1
python resource_crawler.py --id 2   # run a second instance in another terminal
FlagShortRequiredDescription
--id-iYesInstance ID (1-based). Determines state file names and queue claiming.

Download Button Detection

Every resource page has a prominent Download button. The crawler looks for it using these class patterns:

# Regex pattern for the download button
button--(?:resource-download|icon--download)|resourceDownload

The actual HTML typically looks like:

<a class="button button--icon button--icon--download button--cta"
   href="/forum/index.php?resources/slug.59/download">
  <span class="button-text">Download</span>
</a>

External-Only Resources

Some resources are just links to external sites (no downloadable file). These are detected by:

  1. No download button found on the page
  2. Presence of resourceExternal CSS class

External-only resources are skipped from download but still recorded in the manifest with has_file: false and external_only: true.

3 Download Strategies

#StrategyHow It Works
1Download ButtonFind button with class button--icon--download or button--resource-download, extract href.
2"Download" LinkScan for links with "download" text whose href contains resources/ or attachments/.
3/download EndpointFallback: append /download to the resource URL. XFRM always supports this pattern.

Background Thread Pool

Same Phase 1 / Phase 2 pattern as the media crawler. The main thread handles navigation and download URL detection; background workers handle the actual HTTP download and MEGA upload.

Queue Chunking (resource_queue.py)

Same approach as media: categories split into 100-page chunks. Resource categories range from 1 to 109 pages. All 86 categories produce approximately 87 work chunks.

8. Thread Media Crawler

Overview

thread_media_crawler.py is a headless media harvester — no browser required. It processes the pending_media_XX.json queue files generated by crawler.py --text-only, downloading each queued image, attachment, video, avatar, and site asset, then uploading to the exact MEGA path recorded in each queue record.

This enables a two-phase workflow: first crawl all forum text/metadata quickly with --text-only, then download the media files separately (possibly on a faster connection, or in parallel with other crawlers).

CLI Reference

python thread_media_crawler.py --id 1               # process pending_media_01.json
python thread_media_crawler.py --id 1 --max-items 50 # process first 50 pending records
python thread_media_crawler.py --id 1 --dry-run      # report stats without downloading
python thread_media_crawler.py --id 1 --workers 5    # use 5 background threads
FlagShortDefaultDescription
--id-iRequiredInstance ID. Determines which pending_media_XX.json to read.
--max-items-m0 (all)Limit how many records to process in this run.
--dry-run-noffParse queue and show stats without downloading anything. Does not modify files.
--workers-w3Number of background download/upload threads.

State Files

FilePurpose
pending_media_XX.jsonInput — queue of media URLs produced by crawler.py --text-only. Never modified by the harvester.
thread_media_done_XX.jsonDone set — tracks which records have been processed. Prevents duplicate work on restart.
thread_media_manifest_XX.jsonManifest — maps source identity (URL + thread + post) to MEGA path, SHA-256, and status. Used by the mirror builder.
thread_media_progress_XX.jsonProgress — live stats: pending, done, failed counts. Readable by the dashboard.

The queue file is read-only — the harvester tracks completion via the done-set file instead of mutating the queue. This means you can regenerate the queue by re-running crawler.py --text-only without losing progress.

Queue Record Schema

Each record in pending_media_XX.json:

{
  "url":         "https://allthefallen.moe/forum/attachments/image.12345/",
  "type":        "image",          // image | attachment | video | avatar | banner | asset
  "mega_path":   ["Subforum", "Thread_123", "images"],
  "thread_id":   "123",
  "thread_page": 2,
  "post_id":     "post-45678",
  "context":     "",               // e.g. "smilies", "profile:username", "convo:msg_id"
  "queued_at":   "2026-03-08T14:30:00"
}

The unique identity of each record is the tuple (url, type, mega_path, thread_id, thread_page, post_id). The same image URL appearing in two different posts produces two separate records with different placement context.

Typical Workflow

# Phase 1: Fast text-only crawl (no media downloads)
python crawler.py --id 1 --text-only

# Phase 2: Download all queued media
python thread_media_crawler.py --id 1

# Check status
python thread_media_crawler.py --id 1 --dry-run

9. Tools & Utilities

9a. Dashboard (app.py)

app.py is the unified launch & monitoring dashboard. It loads subforum_map.json, divides work across N crawler instances, launches each in its own terminal window, and monitors live progress. Uses customtkinter for a dark-themed GUI. Refreshes every 3 seconds with color-coded status indicators.

The dashboard now shows SQLite tracking stats in the sidebar: total threads done/total, URLs uploaded, sourced directly from crawl_state.db. Instance progress is read from both JSON files and the SQLite progress table.

# Launch the dashboard
python app.py

9b. Archive Viewer (serve_archive.py)

A zero-dependency Python web server (stdlib only: http.server, sqlite3, string.Template) for browsing and searching the downloaded archive.

# Start the viewer
python serve_archive.py /path/to/archive [--port 9000]
# Default port: 8000, requires search.db in same directory

9c. Search Database Builder (build_search.py)

Backfills search.db from existing thread.json files. Walks the directory tree recursively, handles partial saves (thread_partial_*.json), and is safe to re-run (INSERT OR REPLACE).

python build_search.py /path/to/archive

9d. Retry Runner (retry.py)

Re-attempts failed downloads and uploads from the retry queue. Reads crawl_log_{iid}.json files for retry items.

# Retry all instances
python retry.py

# Retry specific instance
python retry.py 3

Handles two item types: image/attachment (download + upload) and upload (file already downloaded, just needs upload). Max 3 retries per item.

9e. Reconciliation Checker (reconcile.py)

Performs a 10-point consistency verification across all state files:

  1. Global .done markers vs crawl log completed threads
  2. Crawl log vs search.db alignment
  3. Downloaded images vs asset manifest
  4. Manifest vs uploaded_urls
  5. Work queue vs crawl log alignment
  6. Retry queue validation
  7. Crawl log stats validation
  8. Cross-instance duplicate detection
  9. Legacy marker detection
  10. Orphaned in_progress threads
# Dry run (report only)
python reconcile.py --dry-run

# Fix issues automatically
python reconcile.py --fix

# Verbose output
python reconcile.py --verbose

# JSON output for scripting
python reconcile.py --json

9f. Repair Tools

repair_failed_images.py

Recovers images that were permanently marked as failed (often due to login page detection bugs). If a thread's failure rate exceeds 15%, it removes failed URLs from dedup and un-completes the thread for re-scraping.

# Report only
python repair_failed_images.py

# Fix issues
python repair_failed_images.py --fix

# Fix specific instance
python repair_failed_images.py --fix --instance 3

repair_zero_threads.py

Finds subforums incorrectly marked as done with 0 threads (due to selector bugs). Un-completes them for retry.

python repair_zero_threads.py --fix

9g. Login Tester (test_login.py)

Opens a visible Playwright browser window to verify that your local auth/session setup works. Checks for "Log in"/"Register" indicators on the page.

python test_login.py

9h. Replica Server (replica/app.py)

replica/app.py is a Flask application that serves a full read-only replica of the forum with XenForo-style query-string routing, original CSS variables, and real saved HTML post bodies. This is the primary way to browse the archive with visual fidelity.

# Start the replica server
cd replica/
python app.py
# Opens at http://localhost:8080

Key differences from serve_archive.py:

  • Route fidelity — mirrors XenForo URL patterns: /index.php?threads/slug.id/page-N, /index.php?forums/slug.id/, direct post links, goto/post, etc.
  • Saved HTML rendering — uses real archived HTML with link/asset rewriting instead of plain-text reconstruction
  • Multi-source data — consumes search.db, crawl_state.db, saved HTML from MEGA, subforum_map.json, and asset manifests
  • Asset serving — local thread images/attachments served via /forum/archive-asset route with manifest-backed path resolution
  • HTML sanitization — strips CSRF tokens, session data, and tracking scripts at render time via data.py

Key files:

FileRole
replica/app.pyFlask app with XenForo-style URL routing (threads, subforums, profiles, search, media, resources, posts)
replica/data.pyData layer — loads all archive sources, parses saved HTML, rewrites links/assets, resolves post locations
replica/templates/Jinja2 templates modeled on XenForo DOM structure (19 templates across pages/ and components/)
replica/static/CSS (xf-variables.css, xf-core.css, replica.css), JS (replica.js), images, fonts

10. Troubleshooting

Files Downloading as .php

Symptom: All downloaded files have .php extension instead of their real type.

Cause: XenForo serves downloads through index.php?resources/slug.59/download. When the server returns Content-Type: application/octet-stream, the old extension detection fell back to parsing the URL and extracted .php from index.php.

Fix: Extension detection now prioritizes Content-Disposition header, and script extensions (.php, .html, .htm, .asp, .aspx, .jsp, .cgi) are filtered from URL fallback.

Login / 2FA Failures

Symptom: Crawler says "Login unable to confirm 2FA" or pages show login forms.

Solutions:

  • Refresh your local auth/session details in config.py
  • Run python test_login.py to verify access before resuming the crawl
  • If the site requires 2FA or an interactive challenge, complete that in the browser first
  • Check the session recovery files only if you need to confirm whether automatic recovery updated them

MEGA Auth Expiry

Symptom: Upload failures with auth errors, "MEGA session expired" in logs.

What happens: The crawler detects auth errors and automatically calls mega_relogin() (protected by mega_lock). If re-login fails, items are added to the retry queue.

If persistent: Check MEGA account status, verify your local access settings in config.py, then run retry.py after fixing.

Rate Limiting (429)

Symptom: Lots of 429 errors in logs, downloads slowing down.

What happens: The crawler respects Retry-After headers and waits the specified duration. Retries with exponential backoff (3s, 6s, 9s).

Mitigation: Reduce the number of concurrent instances or increase the DELAY constant in the crawler config.

HTML Instead of File

Symptom: Downloaded files are actually HTML pages (login forms, error pages).

Detection: download_file() checks the first bytes of the response for HTML indicators (<html, <form, password, "Oops!", "not found"). If detected, the download is rejected.

Common cause: Local session/auth state expired. The server redirected to a login page instead of serving the file.

Stale Queue Leases

Symptom: Work items stuck as "in_progress" but no crawler is working on them.

Automatic fix: After 20 minutes without a heartbeat, the lease is considered stale and another instance can reclaim it.

Manual fix: Edit the queue JSON file directly, change status back to "pending" and clear the owner field.

SQLite Database Issues

Symptom: database is locked errors in logs.

Cause: Too many concurrent writes. WAL mode handles this well but there are limits.

Fix: The busy timeout is set to 5 seconds, which should handle normal contention. If persistent, check that no external tool (DB Browser, etc.) has the database open with a write lock.

Symptom: Migration shows wrong numbers or crawl_state.db seems outdated.

Fix: Re-run python forum/migrate.py --commit. The migration uses INSERT OR IGNORE / ON CONFLICT so it's safe to re-run — existing records won't be overwritten.

Nuclear option: Delete crawl_state.db and re-run the migration. The JSON files are the source of truth during migration. The crawlers will recreate the database on next startup.

General Tips

  • Check crawl_state.db for aggregate stats: python -c "from forum.tracking import db; print(db.get_stats())"
  • Always check crash logs ({prefix}_crash_{iid}.ndjl) or query SELECT * FROM errors ORDER BY timestamp DESC LIMIT 20 for detailed error info
  • Run reconcile.py --dry-run periodically to check for consistency issues
  • Use retry.py after each major crawl session to recover failed items
  • Keep a backup of state files before running repair tools
  • The summary files (summary_{iid}.txt) give a quick human-readable status

11. Operator Notes & Guardrails

Safe Defaults

  • Prefer restarting the same command over hand-editing state files.
  • Run python reconcile.py --dry-run before you assume the crawl is damaged.
  • Use serve_archive.py to inspect real outputs before trying to fix them.
  • Use the repair tools for known failure classes before inventing new manual surgery.

What Usually Deserves Immediate Attention

  • Repeated Could not recover session. messages.
  • Large batches of downloads turning into HTML or login pages.
  • No queue movement across every crawler family at the same time.
  • Unexpected growth in retry queues after a run that looked healthy.

What Often Looks Scary But Is Usually Fine

  • One worker dying while others continue; the lease system can recover that work.
  • Temporary retries, brief rate limiting, or one noisy subforum.
  • Partial archive browsing during an active crawl; the viewer is for inspection, not final parity.
