Archival Museum Design Notes

This document is for people building a replica, archival museum, or long-lived public-facing reconstruction from captured site data. It is not an operator checklist. It is the “why this shape matters” companion to the operator guide and the system reference.

The goal is not simply to save files. The goal is to preserve enough identity, structure, and relationships that the archive can stand in for the original site after the original site is gone.

1. Non-Negotiable Goals

  • Offline-first: the finished archive must still work if the live site disappears entirely.
  • Relationship-preserving: threads, posts, users, media, resources, and categories must still point to each other correctly.
  • Route-faithful: old links should land somewhere meaningful whenever possible.
  • Honest fallbacks: if a thing was not captured, the museum should degrade clearly instead of pretending it has perfect parity.
  • Raw-source preserving: transformed or sanitized output should not replace the original source captures.

2. What Must Be Preserved

EntityWhat matters
Forum / categoryName, hierarchy, description, route, child-forum relationships
ThreadStable ID, title, forum membership, pagination, page count, prefix, last-post info

3. What to Capture While the Site Is Still Alive

  • Representative page HTML for every page type and edge case.
  • Desktop and narrow-width screenshots for visual parity checks.
  • Route patterns and URL variants, including post anchors and old query-string forms.
  • Theme variables, CSS bundles, icons, logos, and local assets.
  • Empty states, error states, permission states, and odd pages like parent forums that do not act like simple thread lists.

Why this matters

Most archive failures are not caused by “missing files.” They are caused by missing relationships and missing UI context: not knowing which page type something belonged to, what link pattern it used, or what state the user expected after a click.

4. Recommended Rendering Strategy

Prefer: saved HTML + structured data + local assets + route fidelity Avoid: rebuilding everything from scratch with a brand-new UX or depending on the live site for missing pieces

5. Fallback Ladder

Different page types can and should use different ladders. Do not force one “universal” content source if the page types are genuinely different.

  • Thread page: saved thread HTML, then thread.json, then search/text fallback.
  • Subforum page: saved forum page HTML, then subforum_index.json, then DB/search-derived rows.
  • Profile page: saved profile HTML/JSON, then lighter metadata fallback.
  • Homepage/sidebar: archive-backed snapshots first, temporary DB-backed fallbacks second, placeholders last.

6. Search, Routing, and Link Fidelity

  • Direct post links need a post_id → thread/page lookup.
  • Quote attribution links must resolve cross-page, not just same-page.
  • Blue inline links inside posts must be rewritten through local route logic and asset manifests.
  • Read-only archive actions should either route sensibly or be visibly non-functional, not broken.

Important lesson

Users notice bad click behavior faster than they notice many visual mismatches. If links feel random, the archive feels untrustworthy even when the saved files are technically present.

Post location lookup chain

Resolving "which page of which thread does post #12345 live on?" requires a two-step lookup:

1. search.db: SELECT thread_id, page_number FROM posts WHERE post_id = ? | found? -> return (thread_id, page) | not found? -> fall through | 2. Saved HTML scan: iterate saved thread pages, parse post IDs from DOM | found? -> return (thread_id, page) | not found? -> 404

8. HTML Sanitization Strategy

Saved HTML from a live site is not display-ready. It contains session-specific data, interactive elements, and tracking artifacts that must be stripped before the museum can render it safely.

What to strip

CategoryExamplesWhy it matters
CSRF tokens<input name="_xfToken">, hidden form fieldsLeaks session state, causes confusing form elements
Session dataLogged-in user indicators, notification badges, "Mark read" linksShows stale session state from crawl time
Tracking scripts<script> tags, analytics beacons, inline JSTries to phone home to dead servers, throws console errors
Interactive formsReply boxes, report buttons, reaction formsNon-functional in read-only archive, confusing to users
Baked-in UI stateXenForo .bbCodeBlock-expandLink "Click to expand" elements, .bbCodeBlock-expandContent wrappersConflicts with dynamic JS that handles expand/collapse based on actual rendered height

Two-stage approach

Our implementation uses two stages: (1) a dedicated sanitize_html.py (887 lines) that runs during crawl time for deep cleaning, and (2) runtime stripping in data.py when parsing saved HTML for display. The runtime stage catches things the crawl-time sanitizer missed or that are only relevant in the replica context (like expand/collapse UI that conflicts with our own JS).

9. Multi-Source Data Model

A museum does not have one universal data source. Different page types consume different combinations of available data, and the "best" source varies by page type.

Page typeSource priority (best → worst)
Thread pageSaved HTML from MEGA → search.db posts (text-only fallback)
Subforum listingSaved subforum HTML → crawl_state.db threads + search.db metadata
Forum homesubforum_map.json + DB stats → dom_structure.json snapshot
User profilecrawl_state.db profile data → username-only placeholder
Searchsearch.db FTS5 (no fallback — requires index)

Why this matters

Trying to force a single "source chain" across all page types leads to either over-engineering (building a universal adapter) or under-serving (some pages always get the worst source). Let each page type pick the best source it has.

10. QA and Parity Tracking

  • Keep a page-type inventory.
  • Track `archive-backed`, `fallback-backed`, and `placeholder` states separately.
  • Keep a mismatch list for real bugs found during visual QA.
  • Measure rich HTML coverage by subforum and by page type.

11. Anti-Patterns to Avoid

  • Do not overwrite raw HTML just because sanitized HTML is easier to work with.
  • Do not collapse parent forums and thread-list pages into one page type.
  • Do not guess missing avatars or other user assets too aggressively.
  • Do not let the museum silently depend on live routes, live CDNs, or live AJAX endpoints.
  • Do not mark placeholders as “done” in documentation just because the shell exists.

13. Lessons Learned

Non-obvious problems encountered during this build that a future museum builder should watch for:

XenForo expand/collapse conflicts

XenForo bakes .bbCodeBlock-expandLink ("Click to expand...") and .bbCodeBlock-expandContent wrapper divs directly into saved HTML. If the museum uses its own JS to handle expand/collapse based on actual rendered height (e.g., a 300px threshold), these baked-in elements cause false "Click to expand" on short quotes. The fix: strip the static elements at render time and let JS be the sole handler.

Scroll offset behind sticky nav

Using scrollIntoView({block:'start'}) puts the target at the very top of the viewport, which gets hidden behind a sticky navigation bar. The fix: calculate scroll position manually using getBoundingClientRect().top + window.scrollY - navHeight - padding. Also set scroll-margin-top in CSS for anchor targets.

URL handler ordering matters

Action links like threads/slug.id/reply&quote=456 match the thread URL pattern before they reach the "dead action link" check. The fix: check for action patterns (react, bookmark, report, reply) before the thread URL handler in the rewrite chain. Order of pattern checks is not just aesthetic — it determines correctness.

Relative vs. absolute link normalization

Saved HTML sometimes contains relative links like index.php?threads/... (no leading /) instead of /index.php?threads/.... All downstream URL handlers expect absolute paths. The fix: normalize early by prepending / to any href starting with index.php? before other rewrite logic runs.

Cross-thread post resolution

When a post redirect resolves via the database, the post might belong to a different thread than the one in the incoming URL (e.g., a cross-thread quote). Using the incoming thread's slug in the redirect URL sends the user to the wrong thread. Always resolve the slug from the database-returned thread ID, not the URL.
